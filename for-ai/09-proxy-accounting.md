# 09. Claude Code через свой прокси + учёт расхода — рецепт для AI-кодера

Цель: пустить Claude Code через **прокси человека** и убедиться, что трафик реально идёт
через него — включая фоновые сессии. Опционально — понять, какой уровень учёта достижим.

Claude Code поддерживает это штатно: [Enterprise network configuration](https://code.claude.com/docs/en/network-config).

---

## 🛑 ПРЕДУСЛОВИЕ — ПРОВЕРИТЬ ДО ПЕРВОЙ КОМАНДЫ

**Этот рецепт — про маршрутизацию через инфраструктуру человека, НЕ про обход страновых
ограничений.**

Anthropic и OpenAI публикуют **закрытые** списки поддерживаемых стран. Дословно у OpenAI:

> **If a location is not included in the list below, our API is not supported there.**

> Accessing or offering access to our services outside of the countries and territories
> listed below **may result in your account being blocked or suspended**.

**Спросить человека и сверить со списками — обе позиции:**

| Что | Anthropic | OpenAI |
|---|---|---|
| Страна, где физически находится человек | [список](https://www.anthropic.com/supported-countries) | [список](https://developers.openai.com/api/docs/supported-countries) |
| Страна, где стоит прокси | тот же | тот же |

- **Обе в списках** → рецепт применим, продолжать.
- **Хотя бы одна отсутствует** → **СТОП.** Не настраивать, не искать обходов, не предлагать
  другие прокси. Прокси не меняет статус страны — он меняет видимую точку выхода. Риск для
  человека: **блокировка аккаунта**, включая оплаченный. Сообщить это и предложить
  [01-claude-code-local.md](01-claude-code-local.md) — Claude Code на локальной Ollama, где
  страновых вопросов нет вовсе.
- **Человек не говорит, где находится** → **СТОП**, спросить. Не додумывать.

## Шаг 0. Определить тип прокси — от этого зависит весь остальной рецепт

Не спрашивать человека «вскрывает ли твой прокси TLS» — он может не знать. **Измерить.**

```bash
ssh <vps> 'ps aux | grep -iE "squid|mitmproxy|nginx|envoy|traefik|litellm|tinyproxy|3proxy" | grep -v grep'
```

Ожидаемый результат: имя ПО прокси. Если пусто — не страшно, решает следующая проверка.

```bash
curl -x http://<vps>:<port> -v https://api.anthropic.com 2>&1 | grep -iE "issuer|subject:"
```

| `issuer` | Тип | Что достижимо по учёту |
|---|---|---|
| Публичный CA (Let's Encrypt, DigiCert, Amazon…) | **A — обычный CONNECT** | только байты и соединения |
| CA человека / его организации | **B — TLS-инспекция** | токены, модель, стоимость |

Если не так (curl не проходит вовсе): прокси недоступен или порт не тот. **Стоп**, уточнить
адрес и порт у человека. Не перебирать порты.

**Записать результат — он определяет, нужен ли `NODE_EXTRA_CA_CERTS` (Шаг 2).**

## Переменные

| Переменная | Значение | Примечание |
|---|---|---|
| `PROXY_URL` | от человека | **только HTTP**, не SOCKS. См. Стоп-условие 2 |
| `HTTPS_PROXY` | `http://<user>:<pass>@<vps>:<port>` | основная |
| `HTTP_PROXY` | то же | запасная, «if HTTPS not available» |
| `NO_PROXY` | `localhost,127.0.0.1` | чтобы локальная Ollama шла мимо прокси |
| `NODE_EXTRA_CA_CERTS` | путь к CA человека | **только для варианта B** |
| Конфиг | `~/.claude/settings.json`, блок `env` | **не шелл** — см. Стоп-условие 3 |

## Шаги

### Шаг 1. Проверить доступность прокси отдельно от Claude Code

```bash
curl -x "$PROXY_URL" -sS -o /dev/null -w "%{http_code}\n" https://api.anthropic.com/v1/messages
```

Ожидаемый результат: любой HTTP-код (**401 — это успех**: прокси пропустил, Anthropic
ответил «нет ключа»). Значит маршрут работает.
Если не так — таймаут или ошибка соединения: **стоп**, дальше не идти. Все последующие
ошибки будут маскироваться под проблемы Claude Code.

### Шаг 2. Только для варианта B — сертификат

```bash
export NODE_EXTRA_CA_CERTS=/path/to/ca.pem
curl -x "$PROXY_URL" --cacert /path/to/ca.pem -sS -o /dev/null -w "%{http_code}\n" https://api.anthropic.com/v1/messages
```

Ожидаемый результат: код ответа, без ошибок TLS.
Если не так: путь к CA неверен или это не тот CA. **Не отключать проверку сертификата**
(`-k`, `NODE_TLS_REJECT_UNAUTHORIZED=0`) — см. Стоп-условие 4.

Путь к CA-файлу берётся **у человека**. Не искать по системе, не генерировать.

### Шаг 3. Записать в settings.json — НЕ в шелл

```json
{
  "env": {
    "HTTPS_PROXY": "http://<user>:<pass>@<vps>:<port>",
    "NO_PROXY": "localhost,127.0.0.1",
    "NODE_EXTRA_CA_CERTS": "/path/to/ca.pem"
  }
}
```

`NODE_EXTRA_CA_CERTS` включать **только для варианта B**.

Ожидаемый результат: файл валиден (`python3 -m json.tool ~/.claude/settings.json`).
Если не так: не ломать существующий `settings.json` — прочитать, добавить блок `env` к тому,
что есть, а не перезаписать файл целиком.

### Шаг 4. Перезапустить супервизор

```bash
claude daemon stop --any
```

Ожидаемый результат: команда отработала. Уже запущенный супервизор **не перечитывает**
конфигурацию — без этого фоновые сессии продолжат ходить со старыми настройками.

### Шаг 5. Проверить интерактивную сессию

```bash
claude -p 'ответь одним словом: ok'
```

Ожидаемый результат: ответ получен, **и запрос виден в логах прокси**.
Если ответ есть, а в логах прокси пусто → **трафик идёт мимо**. Это не успех. См. Шаг 6.

### Шаг 6. 🔥 Проверить ФОНОВУЮ сессию — отдельно

**Этот шаг пропускать нельзя.** Интерактивная сессия может идти через прокси, а фоновая мимо.

```bash
claude --bg -p 'ответь одним словом: ok'
```

Ожидаемый результат: запрос от фоновой сессии **тоже виден в логах прокси**.
Если не так: переменные остались в шелле, а не в `settings.json` (Шаг 3), либо супервизор
не перезапущен (Шаг 4).

## Что достижимо по учёту — сообщить человеку честно

| Вариант | Что видит прокси | Что можно посчитать |
|---|---|---|
| **A** — CONNECT | ничего, туннель зашифрован | байты, соединения, хосты |
| **B** — TLS-инспекция | тело запроса и ответа целиком | модель, токены, стоимость |

🛑 **Не выдавать байты за учёт моделей.** Если у человека вариант A, а он думает, что считает
расход по моделям, — **сказать прямо**. По объёму трафика стоимость не оценивается даже
приблизительно: цена входного и выходного токена различается в разы, а на кэшированный ввод
действует скидка 50–75%.

Учёта ещё нет и его надо строить — **не писать парсер поверх прокси**. Предложить
[LiteLLM](https://docs.litellm.ai/docs/proxy/cost_tracking): токены и стоимость в
`LiteLLM_SpendLogs`, разбивка по ключу/пользователю/команде, UI на `/ui`, виртуальные ключи
с `max_budget`. Требует PostgreSQL и **обязательного `master_key`** (без него — no-auth dev
mode, а расход приписывается именно виртуальному ключу).

**Учёт уже работает и там есть история — не предлагать переезд.** Перенос статистики ради
архитектуры не окупается; это решение человека, а не твоё.

## Стоп-условия

1. **Страна человека или страна прокси отсутствует в списках** — не настраивать, не искать
   обходных путей, не предлагать прокси в другой стране. Предложить локальную Ollama
   ([01](01-claude-code-local.md)). Человек не сказал, где находится — спросить.

2. **Прокси оказался SOCKS** — не пытаться. Дословно: *«Claude Code does not support SOCKS
   proxies.»* У xray обычно подняты оба порта — уточнить у человека HTTP-порт. Не подбирать
   его перебором.

3. **Не полагаться на `export` в шелле.** Супервизор фоновых сессий — один процесс на
   пользователя, наследует окружение первой запустившей его оболочки; как служба — не
   наследует ничего. Дословно: *«...reaches background agents when that shell happened to
   cold-start the supervisor, and **silently doesn't** when a different shell did.»*
   Только `settings.json`.

4. **Никогда не отключать проверку сертификата.** Ни `NODE_TLS_REJECT_UNAUTHORIZED=0`, ни
   `curl -k` как «решение». Ошибка TLS при варианте B означает неверный CA — это чинится
   правильным файлом, а не отключением защиты. Отключив, ты снимаешь единственное, что
   отличает твой прокси от чужого.

5. **Пароль прокси не придумывать и не подбирать** — только от человека. Не коммитить
   `settings.json` с паролем в git: `HTTPS_PROXY` с `user:pass` — это учётные данные.

6. **Путь к CA не искать по системе и не генерировать** — только от человека.

7. **`settings.json` не перезаписывать целиком** — прочитать существующий и добавить блок
   `env`. Там могут быть чужие настройки.

8. **Ответ Claude Code без записей в логах прокси — не успех.** Значит трафик идёт мимо и
   учёт неполный. Доложить, а не закрывать задачу.

## Критерий готовности

```bash
# 1. Прокси доступен (401 — успех: прокси пропустил, Anthropic ответил)
curl -x "$PROXY_URL" -sS -o /dev/null -w "%{http_code}\n" https://api.anthropic.com/v1/messages
```

```bash
# 2. settings.json валиден и содержит HTTPS_PROXY
python3 -m json.tool ~/.claude/settings.json > /dev/null && echo "JSON OK"
grep -q "HTTPS_PROXY" ~/.claude/settings.json && echo "HTTPS_PROXY OK"
```

```bash
# 3. Переменных нет только в шелле — они в файле
grep -q "HTTPS_PROXY" ~/.claude/settings.json || echo "FAIL: только шелл — фон пойдёт мимо"
```

```bash
# 4. В settings.json не утёк пароль в git-репозиторий
git -C ~ check-ignore -q .claude/settings.json 2>/dev/null && echo "игнорируется" || echo "ВНИМАНИЕ: проверить, не под git ли ~/.claude"
```

Требует человека — машиной не проверяется:

- **Обе страны подтверждены по спискам** (Предусловие)
- `claude -p` отвечает **и запрос виден в логах прокси**
- **`claude --bg -p` тоже виден в логах прокси** ← ловит самую дорогую граблю
- Для варианта B: ошибок TLS нет, `NODE_EXTRA_CA_CERTS` задан
- Человеку сказано, какой у него вариант — A или B — и что из этого следует для учёта

Последние два пункта — единственные, которые проверяют то, ради чего всё делалось. Установка,
где Claude Code отвечает, а прокси не видит трафика, проходит все машинные проверки и при
этом бесполезна.
