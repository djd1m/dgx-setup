# 09. Claude Code через свой прокси + учёт расхода моделей

Задача: у тебя есть **свой** прокси-сервер, через который ты хочешь пустить Claude Code —
чтобы считать, сколько уходит на модели. Классический корпоративный сценарий, и Claude Code
поддерживает его штатно.

---

## 🛑 Предусловие. Прочитай, даже если торопишься

**Эта инструкция — про маршрутизацию через собственную инфраструктуру, а не про обход
страновых ограничений.** Разница принципиальная, и она не в технике.

Anthropic и OpenAI публикуют списки стран, где их сервисы работают. Списки **закрытые** —
дословно у OpenAI:

> We do not publish a separate list of countries and territories that we do not support.
> **If a location is not included in the list below, our API is not supported there.**

И последствие:

> Accessing or offering access to our services outside of the countries and territories
> listed below **may result in your account being blocked or suspended**.

**Проверь оба места по спискам, прежде чем что-то настраивать:**

| Что проверить | Anthropic | OpenAI |
|---|---|---|
| Страна, **где физически находишься ты** | [список](https://www.anthropic.com/supported-countries) | [список](https://developers.openai.com/api/docs/supported-countries) |
| Страна, **где стоит твой прокси** | тот же список | тот же список |

**Обе должны быть в списках.** Если да — ты просто ходишь через своё железо, и всё
описанное ниже применимо. Если нет — эта инструкция **не про твой случай**, и прокси
проблему не решает: он меняет видимую точку выхода, а не статус страны. Риск при этом
не «не заработает», а **блокировка аккаунта, включая оплаченный**.

Нужен кодинг-агент без страновых вопросов вообще — он у тебя уже есть:
[01-claude-code-local.md](01-claude-code-local.md), Claude Code на локальной Ollama.
Ни аккаунта, ни ключа, ни списков.

## Шаг 0. Выяснить, что у тебя за прокси

Это решает всё остальное, а помнить такое наизусть никто не обязан.

### Что вообще крутится на сервере

```bash
ssh твой-vps 'ps aux | grep -iE "squid|mitmproxy|nginx|envoy|traefik|litellm|tinyproxy|3proxy" | grep -v grep'
```

### Главный вопрос: вскрывает ли он TLS

От этого зависит, **что ты физически способен посчитать**. Проверяется одной командой —
смотрим, чьим сертификатом представляется `api.anthropic.com`, когда идёшь через прокси:

```bash
curl -x http://твой-vps:8080 -v https://api.anthropic.com 2>&1 | grep -iE "issuer|subject:"
```

| Что видишь в `issuer` | Что это значит |
|---|---|
| Настоящий центр сертификации (Let's Encrypt, DigiCert, Amazon…) | **обычный CONNECT** — прокси не видит содержимое |
| Твой собственный CA (имя твоего сервера/организации) | **TLS-инспекция** — прокси видит весь JSON |

---

## Вариант A. Обычный CONNECT — считаешь байты

Прокси устанавливает туннель и **не видит, что внутри**. TLS идёт насквозь.

**Что можно посчитать:** объём трафика, число соединений, время, к каким хостам ходили.

**Чего посчитать нельзя:** токены, модель, стоимость. Их просто нет в видимости — всё
зашифровано.

⚠️ **И вот важное, если ты думаешь, что уже считаешь модели.** По байтам оценить расход
не выйдет даже приблизительно: у одной и той же модели цена входного и выходного токена
различается в разы, а на кэшированный ввод действует скидка 50–75%. Килобайт трафика не
говорит ничего о деньгах.

Настройка — только переменные из раздела «Общая настройка» ниже. Ничего больше не нужно.

---

## Вариант B. TLS-инспекция — считаешь токены и деньги

Прокси подменяет сертификат своим, расшифровывает, читает, шифрует обратно и отправляет
дальше. Видит тело запроса и ответа целиком — а там и модель, и `usage` с числом токенов.

**Что можно посчитать:** всё. Модель, токены на вход и выход, стоимость, кто ходил.

**Цена вопроса — три вещи:**

**1. Нужно, чтобы Claude Code доверял твоему CA.** Иначе он честно решит, что кто-то
вклинился в соединение, и откажется:

```bash
export NODE_EXTRA_CA_CERTS=/path/to/твой-ca.pem
```

Забудешь — получишь ошибки TLS, которые выглядят как что угодно, кроме «не хватает
сертификата».

**2. Твой прокси видит твои разговоры целиком.** Не только метаданные — весь код, который
ты отправляешь модели, и все ответы. Это не аргумент против, но знать это надо: TLS-инспекция
работает потому, что ты сам себе разрешаешь читать свой трафик.

**3. Разбирать `usage` придётся самому** — прокси даст тебе JSON, но не посчитает деньги.
Если считалка уже написана — отлично. Если нет, см. Вариант C.

## Вариант C. LLM-шлюз — если считалку ещё писать

Если учёта пока нет и его надо строить с нуля, **не пиши парсер поверх прокси**. Готовое
решение — [LiteLLM](https://docs.litellm.ai/docs/proxy/cost_tracking): он не «прокси с логом»,
а шлюз, который понимает LLM-протоколы.

Из коробки: токены и **стоимость в долларах** на каждый запрос в таблицу `LiteLLM_SpendLogs`,
разбивка по ключу / пользователю / команде, веб-интерфейс на `/ui` → вкладка Usage,
эндпоинты `/spend/logs`, `/spend/keys`, `/user/daily/activity`.

И то, чего у прокси не бывает принципиально — **виртуальные ключи с бюджетом**:

```bash
curl 'http://0.0.0.0:4000/key/generate' \
  -H 'Authorization: Bearer <master-key>' \
  -d '{"models": ["gpt-3.5-turbo"], "max_budget": 100, "duration": "30d"}'
```

То есть не «посчитать, сколько потратил», а **упереться в потолок и не потратить лишнего**.
Требует PostgreSQL. Подробности про сам LiteLLM — в [02-claude-code-cloudru.md](02-claude-code-cloudru.md),
там он в роли переводчика протокола.

> ⚠️ **Если берёшь LiteLLM ради учёта — `master_key` обязателен.** Без него он работает
> в режиме «no-auth dev mode» и ключ не проверяет; а расход приписывается **именно
> виртуальному ключу**. Нет мастер-ключа → нет виртуальных ключей → некому приписать трату.

**Когда C не нужен:** если учёт у тебя уже работает и там накоплена статистика — не переезжай.
Переносить историю ради красоты архитектуры не стоит.

## Общая настройка (нужна во всех вариантах)

```bash
export HTTPS_PROXY=http://user:password@твой-vps:8080
```

Пароль в переменной — плохо; документация [прямо предупреждает](https://code.claude.com/docs/en/network-config):
*«Avoid hardcoding passwords in scripts»*. Если прокси только твой и слушает не наружу —
лучше вообще без авторизации, чем пароль в открытом виде.

Обойти прокси для локальных адресов (Ollama, например):

```bash
export NO_PROXY="localhost,127.0.0.1"
```

`HTTP_PROXY` тоже поддерживается — как запасной вариант, «if HTTPS not available».

## Три грабли

### 1. 🛑 SOCKS не поддерживается

Дословно:

> **Claude Code does not support SOCKS proxies.**

Если у тебя xray, там обычно подняты оба: SOCKS5 на одном порту, HTTP на соседнем.
**Бери HTTP-порт.** Укажешь SOCKS — не заработает, и сообщение будет невнятным.

### 2. 🔥 Фоновые агенты не увидят твой `export`

Самая дорогая грабля, потому что отказ **тихий**. Дословно из документации:

> If you export a proxy, CA path, or mTLS variable only in your shell, it reaches background
> agents when that shell happened to cold-start the supervisor, and **silently doesn't** when
> a different shell did.

Причина: супервизор фоновых сессий — **один процесс на пользователя**. Он наследует окружение
той оболочки, которая запустила его первой; установленный как служба — не наследует ничего.

В терминале работает, в фоне — молча мимо прокси и мимо учёта. **Клади в `~/.claude/settings.json`:**

```json
{
  "env": {
    "HTTPS_PROXY": "http://user:pass@твой-vps:8080",
    "NODE_EXTRA_CA_CERTS": "/path/to/ca.pem",
    "NO_PROXY": "localhost,127.0.0.1"
  }
}
```

Это единственный способ, достающий до **всех** сессий. Уже поднятый супервизор конфигурацию
не перечитает — после правки `claude daemon stop --any`.

### 3. Одного `api.anthropic.com` мало

Откроешь только его — не залогинишься:

| Хост | Зачем |
|---|---|
| `api.anthropic.com` | запросы к модели |
| `claude.ai` | аутентификация |
| `platform.claude.com` | вход через Console |
| `downloads.claude.ai` | плагины, автообновление |
| `raw.githubusercontent.com` | `/release-notes` |

Полный список — в [документации](https://code.claude.com/docs/en/network-config).
