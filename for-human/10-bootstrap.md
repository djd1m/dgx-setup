# 10. Один скрипт: чистый DGX → Claude Code через прокси в КЗ + учёт

Сценарий: у тебя **новый DGX** (например, взял в командировку), и надо, чтобы на нём
заработал Claude Code, который ходит в модели **Anthropic и OpenAI через твой прокси в
Казахстане**, а расход считался в **LiteLLM**. Одной командой: скрипт сам смотрит, что уже
стоит, и доставляет недостающее.

Это «пусковой» слой. Дальше можно дать AI-кодеру ссылку на [`for-ai/`](../for-ai/) — и он
доставит Ollama, агентов, навыки уже сам.

> Скрипт — не замена [09-proxy-accounting.md](09-proxy-accounting.md), а его автоматизация.
> Почему именно такая цепочка и почему x-ui считает только байты — там. Здесь — как запустить.

---

## 🛑 Сначала — предусловие про географию

Оно же в [09](09-proxy-accounting.md), но повторю, потому что скрипт его не отменяет:

**И страна, где физически стоит DGX, И страна прокси (КЗ) должны быть в списках
поддерживаемых** — у [Anthropic](https://www.anthropic.com/supported-countries) и у
[OpenAI](https://developers.openai.com/api/docs/supported-countries). Прокси меняет видимую
точку выхода, **а не статус страны**. Твой рабочий случай (DGX в Амстердаме + прокси в КЗ)
проходит: обе страны в обоих списках. Если однажды это будет не так — риск не «не заработает»,
а **блокировка аккаунта**, включая оплаченный. Скрипт для обхода страновых запретов не нужен
и не предназначен.

---

## Что должно быть под рукой

| Что | Зачем | Если нет |
|---|---|---|
| Открытый интернет **на время установки** | скачать xray, Claude Code, образ LiteLLM | ставь до вылета / там, где Cloudflare не режут |
| `vless://…` ссылка на твой VPS в КЗ | из неё скрипт соберёт туннель | возьми в своей панели/клиенте, кнопка «Share» |
| Ключ Anthropic и/или OpenAI | их LiteLLM использует как исходящие | хотя бы один обязателен |
| Docker Engine для aarch64 | в нём поднимется LiteLLM + PostgreSQL | [инструкция Docker для Ubuntu](https://docs.docker.com/engine/install/ubuntu/) |

> ⚠️ **Установщик Claude Code гео-блокируется — идёт через туннель.** `claude.com` в некоторых
> регионах отдаёт страницу «App unavailable in region» вместо `install.sh` (и с виду это HTTP 200,
> поэтому скрипт раньше пытался её выполнить и падал с «синтаксической ошибкой рядом с `<`»).
> Теперь фаза Claude тянет установщик и бинарь **через туннель** (с поддерживаемого IP) и
> проверяет, что скачан скрипт, а не HTML. Прочие пакеты (xray с GitHub, образ LiteLLM с ghcr.io)
> гео не блокируются — их через КЗ гнать не надо; их режет только российский DPI из-за Cloudflare
> ([09 объясняет](09-proxy-accounting.md)). Через КЗ-прокси идёт трафик моделей **и** установщик Claude.

## Что собирает скрипт

```
Claude Code
   │  ANTHROPIC_BASE_URL=http://127.0.0.1:4000   (локально, без прокси)
LiteLLM :4000            считает модель, токены, стоимость, бюджеты
   │  HTTPS_PROXY=http://127.0.0.1:10809         свой выход — в туннель
xray-клиент :10809       транспорт: HTTP-inbound → VLESS
   │  VLESS
твой VPS в КЗ            продолжает считать байты, история цела
   │
api.anthropic.com / api.openai.com
```

Важное в этой схеме — **два слоя не конфликтуют**. LiteLLM даёт токены и деньги (чего в
туннеле нет физически), а VPS остаётся как есть, со всей накопленной статистикой. Ничего не
переносишь и не переразворачиваешь.

И обрати внимание: Claude Code смотрит на LiteLLM через **`ANTHROPIC_BASE_URL`**, а не через
`HTTPS_PROXY`. LiteLLM — это API-шлюз, а не forward-прокси; путать эти две вещи — типовая
ошибка (в туннель через `HTTPS_PROXY` уходит уже сам LiteLLM).

## Запуск

```bash
git clone https://github.com/djd1m/dgx-setup.git
cd dgx-setup

# 1. сначала посмотреть, что скрипт видит на машине (ничего не ставит):
bash scripts/dgx-claude-bootstrap.sh --diagnose

# 2. если картина ок — полный проход:
bash scripts/dgx-claude-bootstrap.sh
```

> В России `raw.githubusercontent.com` и часть GitHub за Cloudflare режутся — поэтому
> `curl | bash` одной строкой не предлагаю: до сырого файла трафик может не дойти. `git clone`
> надёжнее, а на открытой сети (Амстердам) и подавно всё работает.

Скрипт задаст ровно то, что нельзя угадать: **vless-ссылку** (ввод скрыт, как пароль) и
**ключи** Anthropic/OpenAI. Всё это ложится в `~/.dgx-claude/secrets.env` с правами `600` —
и **никуда больше**: ни в репозиторий, ни в чат, ни в логи.

### Полезные флаги

| Флаг | Что делает |
|---|---|
| `--diagnose` | только диагностика, ничего не устанавливает |
| `--skip-xray` | туннель уже поднят — не трогать |
| `--skip-litellm` | учёт уже есть — не поднимать второй |
| `--yes` | не переспрашивать (для повторных прогонов) |

## Что происходит по фазам

1. **Диагностика.** `uname -m` (ждёт `aarch64`), ОС, память, что уже стоит (Claude Code,
   docker, xray), свободны ли порты `10809 / 4000 / 5432`, тянется ли GitHub / ghcr.io /
   downloads.claude.ai. Ничего не меняет.
2. **Туннель.** Качает xray (**с проверкой SHA256** — не совпал, скрипт останавливается),
   собирает клиентский конфиг из твоей ссылки, поднимает его как `systemd --user`
   (переживает выход из сессии) и **проверяет, что выход — Казахстан** (`ipinfo.io/country`
   → `KZ`). Не KZ — предупредит и спросит.
3. **Claude Code.** Официальный установщик, при недоступности — через `npm`.
4. **LiteLLM + PostgreSQL.** Поднимает `docker compose` с запиннеными образами
   (`litellm:v1.92.0`, есть arm64; `postgres:16-alpine`). Мастер-ключ и пароль БД
   генерирует сам. Исходящие LiteLLM направляет в туннель.
5. **Привязка.** Пишет `~/.claude/settings.json` (блок `env`), не затирая остальное.
   Кладёт именно в файл, а не в шелл — иначе [фоновые агенты не увидят настройку](09-proxy-accounting.md)
   (грабля №2 из 09).
6. **Проверка.** Три теста, каждый **может честно упасть**: туннель выходит из KZ; LiteLLM
   отвечает на `/health/liveliness`; и главное — **живой запрос к модели** через всю цепочку
   возвращает HTTP 200. Если что-то не так — покажет, что именно.

## Готово, если

- [ ] `--diagnose` показывает `aarch64` и свободные порты
- [ ] туннель отвечает и выход — `KZ`
- [ ] `claude --version` работает
- [ ] `http://127.0.0.1:4000/health/liveliness` отвечает
- [ ] финальный живой запрос к модели вернул `HTTP 200`
- [ ] расход виден в UI LiteLLM: `http://127.0.0.1:4000/ui` (ключ — `LITELLM_MASTER_KEY` из `secrets.env`)
- [ ] на VPS в x-ui **растут счётчики байт** во время сессии Claude Code — значит трафик реально идёт через КЗ, а не мимо

Последний пункт — тот самый ground truth: LiteLLM показывает токены и деньги, а x-ui на VPS
показывает, что байты действительно прошли через твой канал.

---

## Если что-то пошло не так

| Симптом | Куда смотреть |
|---|---|
| туннель не выходит наружу | `journalctl --user -u dgx-xray -n 50` — чаще всего нерабочая ссылка |
| выход не из KZ | ссылка ведёт на другой сервер, чем ты думал |
| LiteLLM не поднялся | `docker logs dgx-litellm` — обычно опечатка в ключе или занятый порт 4000 |
| Claude Code не видит настройку в фоне | `claude daemon stop --any`, затем перезапуск (грабля №2 из [09](09-proxy-accounting.md)) |
| хочется убедиться, что LiteLLM реально уходит в туннель | смотри счётчики x-ui на VPS во время запроса — если растут, трафик идёт через КЗ |

## Что здесь **не** проверено (честно)

- **Что LiteLLM гарантированно уважает `HTTPS_PROXY` для исходящих.** LiteLLM ходит через
  `httpx`, а тот по умолчанию читает `HTTPS_PROXY` из окружения — но версии меняются.
  Единственная надёжная проверка — счётчики x-ui на VPS (см. чек-лист). Проверь их на первом
  запросе.
- **Что `downloads.claude.ai` доступен именно с твоей машины.** Если официальный установщик
  не тянется — скрипт сам откатывается на `npm`, но `npm` должен быть в системе.
- **Точные id моделей** в `litellm.config.example.yaml` — шаблонные. Сверься со списками
  [Anthropic](https://docs.claude.com/en/docs/about-claude/models/overview) и
  [OpenAI](https://platform.openai.com/docs/models) и поправь под актуальные.
- **Что именно у тебя на VPS в КЗ — VLESS.** Скрипт исходит из этого (как на инспектированном
  сервере). Если там **настоящий HTTP-прокси** (не VLESS), туннель-фаза не нужна: запусти
  `--skip-xray` и укажи прокси напрямую, как в [09, Вариант A/C](09-proxy-accounting.md).
